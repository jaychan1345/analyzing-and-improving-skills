# 详细中英文 README 改版设计

日期：2026-07-16  
状态：用户已确认  
适用仓库：`analyzing-and-improving-skills`

## 1. 背景

当前根目录 `README.md` 只有简短英文概述、五个运行时 Skill 的一句话说明和安装文档链接。它能够说明仓库“是什么”，但不能帮助第一次接触本项目的同事理解以下问题：

- 五个 Skill 分别在什么情况下调用；
- 拆解、优化、评测和治理之间如何衔接；
- 为什么优化操作需要审批，而只读拆解不需要；
- 中央版本、改进提案和个人分支之间是什么关系；
- 跨境电商团队如何把 ERP 表格、财务对账和标准操作流程应用到本体系；
- 如何安装、验证、升级、贡献和排查常见问题。

因此，README 需要从“仓库简介”升级为“团队级使用入口”。

## 2. 已确认方案

采用方案 C：团队级操作手册。

- `README.md` 作为详细中文版和默认入口；
- 新增 `README_EN.md`，提供独立英文说明；
- 两份文档顶部提供语言切换链接；
- `INSTALL.md` 继续作为独立安装手册，README 只保留快速安装和安装文档入口；
- 中文版优先服务公司内部非技术人员，同时保留技术人员需要的目录、测试、版本和贡献说明。

## 3. 目标读者

### 3.1 主要读者

- 跨境电商运营人员；
- 从 ERP 下载报表并执行标准分析流程的数据人员；
- 下载对账单并输出标准财务账单的财务人员；
- 负责整理、维护或分享内部 Skill 的流程负责人；
- 审核中央 Skill 改进提案的维护者。

### 3.2 次要读者

- 安装本 Skill 套件的个人用户；
- 希望维护个人分支的使用者；
- 需要检查 Skill 质量、回归风险或发布状态的评测人员。

## 4. 文档架构

### 4.1 `README.md` 中文主文档

按以下顺序组织：

1. 项目定位与适用范围；
2. 适合解决和不适合解决的问题；
3. 五个 Skill 的职责、输入、输出和边界；
4. 请求如何在五个 Skill 之间路由；
5. 完整工作流：拆解、提出改进、批准、生成派生版本、评测、发布；
6. 审批式迭代机制；
7. 中央版本与个人分支规则；
8. 跨境电商典型案例；
9. 仓库目录结构；
10. 快速安装、验证和升级；
11. 常用提示词示例；
12. 输出物与证据要求；
13. 测试、发布和回滚；
14. 常见问题与故障排查；
15. 贡献和维护建议；
16. 版本信息与许可证说明。

### 4.2 `README_EN.md` 英文文档

英文版与中文版保持相同的核心事实和操作规则，但语言更紧凑。它至少覆盖：

- suite purpose；
- five-Skill responsibility matrix；
- routing and approval model；
- installation and verification；
- repository structure；
- usage examples；
- contribution and governance links。

英文版不得引入中文版中不存在的新治理规则。

### 4.3 `INSTALL.md`

保持安装细节的唯一权威来源。README 中只展示最短可执行路径，并链接到 `INSTALL.md`。若 README 与 `INSTALL.md` 不一致，以修复两者为准，不在 README 中声明其中一份自动覆盖另一份。

## 5. 核心信息设计

### 5.1 五个 Skill 的职责边界

README 需要明确展示：

| Skill | 核心职责 | 是否修改目标 Skill | 典型产物 |
|---|---|---:|---|
| `analyzing-skills` | 只读拆解、审计和反向分析 | 否 | 分析简报、证据表、问题清单 |
| `improving-skills` | 经批准后修改、拆分、重构或生成派生版本 | 是 | 变更计划、派生 Skill、差异说明 |
| `evaluating-skills` | 测试触发、行为、失败路径、回归和发布准备度 | 否，除非另行批准 | 测试矩阵、评测报告、发布结论 |
| `governing-skill-evolution` | 管理提案、审批、版本、个人分支、合并和回滚 | 管理变更，不直接替代实现流程 | 改进提案、审批记录、版本记录 |
| `orchestrating-skill-work` | 编排跨越两个及以上阶段的请求 | 取决于下游阶段 | 阶段计划、交接物、最终汇总 |

### 5.2 路由逻辑

README 采用简洁的决策顺序：

1. 只要求理解、拆解或审计：使用 `analyzing-skills`；
2. 明确要求改动或生成派生版本：进入 `improving-skills`；
3. 要求验证、比较或判断能否发布：使用 `evaluating-skills`；
4. 要求修改本套拆解体系本身：使用 `governing-skill-evolution`；
5. 一个请求跨越两个或更多阶段：先使用 `orchestrating-skill-work`。

README 必须强调：“分析发现问题”不等于“自动获得修改授权”。

### 5.3 审批式迭代

中央套件默认只接受改进提案：

1. 使用者发现问题；
2. 提交结构化改进提案；
3. 维护者判断接受、拒绝、要求补充或推迟；
4. 获批后才修改中央版本；
5. 修改后必须评测；
6. 通过发布检查后才合并或发布。

使用者只有在明确选择“维护个人分支”后，才能脱离中央审批节奏维护自己的变体。个人分支不自动回写中央版本。

### 5.4 跨境电商案例

至少提供三个可识别但不绑定具体 ERP 品牌的案例：

- 运营：将 ERP 导出的销售、库存和广告表格操作流程封装为 Skill；
- 财务：将平台对账单清洗、字段映射和标准账单输出流程封装为 Skill；
- Skill 维护：收到外部 Skill 后，先拆解证据和依赖，再提交适配本公司的改进提案。

每个案例说明推荐路由、关键输入、风险点和预期输出，不虚构业务阈值或财务口径。

## 6. 使用体验

README 应满足两类阅读路径：

- 新用户可在前半部分完成“理解套件—选择 Skill—安装—第一次调用”；
- 维护者可在后半部分找到“审批—评测—版本—发布—回滚—贡献”规则。

文档采用结论优先、短段落、表格和可复制提示词。避免把 `SKILL.md` 内容逐字复制进 README；详细规则通过相对链接指向对应文件。

## 7. 错误与边界处理

README 需要明确以下情况：

- 输入 Skill 只有 `SKILL.md`、但引用文件缺失时，分析结果必须标记为不完整；
- 用户只授权拆解时，不得修改源文件；
- 用户要求优化但未确认修改范围时，只能提交提案或变更计划；
- 评测失败时不得声明“可发布”；
- 中央版本的修改必须经过审批和回归测试；
- 个人分支的变更不代表中央版本的官方规则；
- 文档中的示例阈值、字段和文件名均为示意，实际业务口径由使用者提供。

## 8. 验证策略

实施完成后执行以下检查：

1. 确认 `README.md` 为中文主文档，且存在指向 `README_EN.md` 的语言链接；
2. 确认 `README_EN.md` 能返回中文主文档；
3. 检查所有仓库内相对链接的目标文件存在；
4. 检查五个 Skill 的名称、职责和目录与仓库实际内容一致；
5. 检查 README 描述的审批与个人分支规则和治理参考文件一致；
6. 检查安装命令与 `INSTALL.md` 一致；
7. 扫描 `TODO`、`TBD`、占位符和失效路径；
8. 检查工作树只包含本次文档改动；
9. 在提交前审阅最终差异；
10. 推送新分支并通过 PR 合并，不直接覆盖已有发布历史。

## 9. 非目标

本次改版不做以下变更：

- 不修改五个运行时 Skill 的触发或执行逻辑；
- 不修改审批、分支或发布政策；
- 不新增业务计算脚本；
- 不绑定具体 ERP、平台或财务软件；
- 不把治理测试目录安装为运行时 Skill；
- 不重新设计 Skill 套件的文件结构。

## 10. 验收标准

完成后的文档应满足：

- 中文读者无需阅读英文即可理解并使用完整套件；
- 新用户能够在 README 中正确选择五个 Skill 之一或进入编排流程；
- 使用者能够清楚区分只读分析、获批优化和体系治理；
- 团队成员知道默认只能提交中央改进提案，也知道如何明确选择个人分支；
- 跨境电商运营和财务人员能将自己的标准表格流程映射到本体系；
- README 与实际目录、安装说明和治理规则不存在可识别冲突；
- 两种语言均不存在占位内容、死链或错误 Skill 名称。

## 11. 交付位置

同一版本的中英文 README 必须同步到以下位置：

- 本地 Git 仓库文件夹；
- 供团队成员传阅的本地 ZIP 包；
- GitHub 仓库默认分支。

发布完成后分别验证本地文件内容、ZIP 内文件清单，以及 GitHub 默认分支上的 README 提交版本。不得只更新其中一个位置。
